# Railway environment variables — Supabase JWT gateway

Configure the **backend gateway** on **Railway** so `Authorization: Bearer <supabase_access_token>` verifies correctly.

**Do not paste real secrets into chat or commit them to git.** Set values only in **Railway → service → Variables** (mark secrets as **Secret**). Never hardcode them in the repo.

---

## Copy-paste prompt (for Cursor / any coding agent)

```text
You are configuring Railway environment variables for a backend gateway that verifies Supabase user access tokens (Bearer JWT from the browser).

Goals:
1. List every env var related to JWT / Supabase auth the gateway needs.
2. For each var, say: purpose, where to copy it from (Supabase Dashboard path), and whether it is secret.
3. Use placeholders only (e.g. <SUPABASE_JWT_SECRET>) — never output or guess real values.
4. Confirm issuer (`iss`) claim expectations if the gateway checks `SUPABASE_JWT_ISS`.

Gateway auth stack (Manthana pattern):
- `JWT_SECRET`: long random string (≥32 chars), generated by us for the gateway — NOT from Supabase; required for startup / legacy paths.
- `SUPABASE_JWT_SECRET`: exact string from Supabase → Project Settings → API → **JWT Secret** (shared signing secret for HS256 access tokens in the default setup). Railway must match this or Bearer validation fails.
- `SUPABASE_JWT_ISS` (optional): if the gateway enforces issuer, set to `https://<project-ref>.supabase.co/auth/v1`.

Operational checks after deploy:
- `GET <gateway>/health` → 200 without auth.
- `GET <gateway>/v1/health` with `Authorization: Bearer <valid Supabase session access_token>` → 200.

If tokens fail: verify clock skew, that the token is an **access** token (not opaque refresh string), that `SUPABASE_JWT_SECRET` was copied with no trailing newline/spaces, and that the Supabase project ref matches the frontend’s `NEXT_PUBLIC_SUPABASE_URL`.
```

## Environment variables required

| Variable | Purpose | Source | Secret? |
|----------|---------|--------|-----------|
| `JWT_SECRET` | Gateway startup requirement; legacy/internal auth paths. **Not** Supabase-issued. Must be **≥32** random characters. | Generate locally (see below) | Yes |
| `SUPABASE_JWT_SECRET` | Verifies signatures on Supabase-issued **access** tokens (HS256 default). Must match Supabase’s signing key. | Supabase Dashboard → Project Settings → API → **JWT Secret** (copy exactly, no trailing whitespace) | Yes |
| `SUPABASE_JWT_ISS` | If the gateway validates the JWT `iss` claim, set to the Supabase Auth issuer URL. | Construct: `https://<project-ref>.supabase.co/auth/v1` | No |
| `SUPABASE_URL` | **Optional but recommended for asymmetric JWTs** — project API URL (`https://<project-ref>.supabase.co`). The gateway derives **JWKS** (`…/auth/v1/.well-known/jwks.json`) and default **issuer** when JWT Signing Keys (RS256/ES256) are enabled. If omitted, the gateway can still derive JWKS from token `iss` for Supabase-issued access tokens. | Supabase Dashboard → Project Settings → API → **Project URL** | No |

## How to locate each value

### `SUPABASE_JWT_SECRET` (critical)

1. Log in to [Supabase Dashboard](https://supabase.com/dashboard).
2. Select your project.
3. **Project Settings** (gear, sidebar) → **API**.
4. Under **JWT Settings**, find **JWT Secret**.
5. Copy the **entire** string (long random/base64-like value).
6. Paste into Railway with **no** trailing spaces or newlines.

### `SUPABASE_JWT_ISS` (optional, recommended if you enforce `iss`)

1. Same path: **Project Settings** → **API**.
2. Note **Project URL** or reference id (the subdomain before `.supabase.co`), e.g. `drhcreysmpomaiypcsjy`.
3. Set: `https://<project-ref>.supabase.co/auth/v1` (example: `https://drhcreysmpomaiypcsjy.supabase.co/auth/v1`).

### `JWT_SECRET` (generate locally)

Do **not** copy from Supabase. Generate a new random string:

```bash
# Option 1: OpenSSL
openssl rand -base64 32
```

```bash
# Option 2: Python
python -c "import secrets; print(secrets.token_urlsafe(32))"
```

Option 3: password manager or Railway’s secret generator.

## Critical distinctions

| Item | Role | Use on gateway as `SUPABASE_JWT_SECRET`? |
|------|------|------------------------------------------|
| **JWT Secret** (API → JWT Settings) | Signs/verifies JWTs issued by Supabase | **Yes** — this is what you copy to `SUPABASE_JWT_SECRET`. |
| **Anon** / **publishable** key (Project API keys) | Public client key for browser → Supabase | **No** — not the JWT signing secret for HS256 verification. |
| **Service role** key | Bypasses RLS; trusted server only; never expose to browsers | **No** — not for verifying end-user Bearer tokens on the gateway. |

## Verification checklist after deploy

### 1. Health (no auth)

```bash
curl -i "https://<your-gateway>.railway.app/health"
```

Expected: **200 OK**.

### 2. Health (valid Supabase access token)

```bash
curl -i -H "Authorization: Bearer <valid_access_token>" \
  "https://<your-gateway>.railway.app/v1/health"
```

Expected: **200 OK**.

**Getting a test token**

- Sign in through your **frontend** (Supabase client), then read **`session.access_token`** from app state or DevTools (Application → Local Storage / session storage, depending on your auth storage).
- Ensure it is the **access** token (JWT), not the **refresh** token (opaque string).

## Troubleshooting token failures (401 / 403 on `/v1/health`)

| Issue | Check |
|-------|--------|
| Signature invalid | `SUPABASE_JWT_SECRET` matches Dashboard **exactly** (no spaces/newlines). |
| Token expired | Use a **fresh** access token (~hourly expiry is common). |
| Clock skew | Container time; logs mentioning `exp` / `nbf`. |
| Wrong token type | Must be **access** JWT, not refresh token. |
| Project mismatch | Frontend `NEXT_PUBLIC_SUPABASE_URL` project ref matches the project whose **JWT Secret** you configured. |
| Issuer mismatch | If gateway checks `iss`, `SUPABASE_JWT_ISS` must be exactly `https://<project-ref>.supabase.co/auth/v1`. |

## Secret rotation (Supabase JWT Secret)

If Supabase **rotates** the JWT secret:

1. Update **`SUPABASE_JWT_SECRET`** on Railway **immediately**.
2. Existing tokens signed with the old secret fail until users **re-authenticate** (new tokens).
3. Prefer rotating in a **low-traffic** window with a communicated maintenance note if needed.

---

## Railway configuration steps

1. Open Railway → your **gateway** service → **Variables**.
2. Add:
   - `JWT_SECRET` = generated value (**mark as Secret**).
   - `SUPABASE_JWT_SECRET` = value from Supabase Dashboard (**mark as Secret**).
   - `SUPABASE_JWT_ISS` = `https://<project-ref>.supabase.co/auth/v1` (optional; usually not marked secret).
3. **Deploy**, then run the [verification](#verification-checklist-after-deploy) curls above.
